Cómo usar el terminal de producción

Pensado para operarios y supervisores de planta. No necesita conocimientos informáticos.

1. Iniciar su turno

Acerque su fotocheck al lector hasta oír un pitido corto: la sesión se abre sola. Si el lector no responde, escriba su PIN en el teclado grande de la pantalla y pulse el botón verde de confirmación. No hace falta usar el teclado del teléfono.

2. Registrar producto terminado

Escanee el código de cada producto que sale de la línea. Cada lectura correcta suena con un pitido y vibra; el número grande del centro sube solo y muestra cuántas unidades lleva el pallet actual.

3. Qué significan los colores

ColorQué indicaQué hacer
VerdeLectura correcta.Continúe escaneando con normalidad.
Amarillo (barra superior)No hay internet en este momento. El terminal guarda todo localmente.Siga trabajando igual. Se enviará solo cuando vuelva la señal.
RojoHubo un problema (lectura inválida, fallo, o sincronización bloqueada).Si aparece el botón grande, pulse "Reintentar". Si la barra roja pide supervisor, avise a su supervisor.

4. Corregir una lectura equivocada

Debajo del número grande aparecen las últimas lecturas. Si escaneó algo por error, pulse el botón rojo de la papelera junto a esa lectura para quitarla. El contador se ajusta solo.

5. Cerrar el pallet y enviarlo a bodega

Cuando termine el pallet, pulse el botón ancho de abajo "Concluir Traslado". El terminal traspasa la custodia del lote hacia la bodega central, vuelve a cero y queda listo para el siguiente pallet.

6. Cerrar sesión

Al terminar su turno pulse "Salir" arriba a la derecha. El siguiente compañero podrá identificarse con su fotocheck o su PIN.

Sonidos: pitido agudo = todo bien. Zumbido grave y doble vibración = algo falló (revise el color de la pantalla).

Manual técnico · Administración y soporte

Stack y arquitectura

LAMP nativo (PHP 8.x con declare(strict_types=1), sin frameworks). index.php renderiza condicionalmente según la presencia de config.php e inyecta window.__OMNI_BOOTSTRAP__ como fuente de verdad estructural. El frontend es JS Vanilla modular sin Vue/React.

ArchivoResponsabilidad
index.phpDOM, render condicional, bootstrap servidor→cliente, modales (incluye el visor de manuales por iframe).
setup.phpBackend POO del wizard: proxy de mismo origen al API CORE y escritura de config.php.
setup.jsFrontend del asistente paso a paso.
api-client.jsCliente operativo: login, catálogo, conversión metrológica, traspasos atómicos.
outbox-service.jsCola Outbox offline en IndexedDB y sincronización diferida.
app.jsOrquestador: router, escáner, ciclo de producción, banners, anti-crash.
manual.htmlEsta documentación (autónoma, abrible sin desplegar).
config.phpAutogenerado por setup.php (no versionar). Sólo identificadores estructurales.

Autoinstalación (Setup Wizard)

El asistente sólo aparece si no existe config.php. setup.js consume setup.php?action=… (mismo origen, sin CORS) para: test de conexión, login de supervisor, descarga de interlocutores/ubicaciones e instalación. Al concluir, setup.php escribe config.php con chmod 0600, el cliente cachea la config cifrada y la app se auto-reinicia (recarga) hacia el login.

Re-despliegue / purga de nodo

  • Eliminar config.php del servidor y limpiar el almacenamiento del navegador (claves omni1004.* en localStorage e IndexedDB omni1004_outbox).
  • O bien POST setup.php con { "action":"purge", "token":"<JWT supervisor>" }.

Patrón Outbox y persistencia

Las conclusiones de pallet se encolan en IndexedDB (omni1004_outbox) con su timestamp físico. Un servicio asíncrono sincroniza en orden cronológico estricto al volver la red.

  • Microcorte (error de red): pausa y reintento automático cada 20 s y al evento online.
  • Rechazo lógico (HTTP 4xx): congelamiento persistente, banner rojo, desbloqueo sólo por supervisor.

Comunicación con el [1001] API CORE

Operación directa por fetch con Authorization: Bearer <JWT>. Stock siempre como delta atómico relativo (jamás SET absoluto). Conversión metrológica a unidad base (g/ml/ud) en la frontera de envío. El traspaso usa el tipo TRANSFER_CUSTODY (custodia → tránsito) con idempotency_key + occurred_at.

Contrato de endpoints (ajustar a las rutas reales del v6, centralizado en api-client.js y setup.php):

GET  /health
POST /auth/login              { identifier, pin } -> { token, user }
GET  /interlocutors
GET  /locations?interlocutor_id=
GET  /products?interlocutor_id=
POST /inventory/transfers     { type, idempotency_key, from_location_id,
                                to_location_id, occurred_at, items:[{sku,delta,unit}] }

Manual de despliegue

Instalación del terminal en hosting compartido (LAMP) o entorno local XAMPP. Subdominio de referencia: https://produccion.omni.josepan.app.

1. Requisitos del servidor

  • PHP 8.0 o superior con extensión curl habilitada (alternativamente allow_url_fopen=On para el fallback por streams).
  • Apache con mod_rewrite (recomendado) y HTTPS válido en el subdominio.
  • Permiso de escritura del usuario web sobre el directorio raíz del nodo (para generar config.php y setup.log).

2. Subir archivos

Copiar al document root del subdominio los siguientes archivos (todos en el mismo directorio):

produccion.omni.josepan.app/
├── index.php
├── setup.php
├── setup.js
├── app.js
├── api-client.js
├── outbox-service.js
├── manual.html
└── .htaccess        (ver punto 4)
# config.php y setup.log se crean solos en el primer arranque

3. Permisos

# El usuario de Apache (www-data / nobody) debe poder escribir en el directorio:
chmod 755 .                 # directorio del nodo
# Tras la instalación, config.php queda en 0600 automáticamente.
# Si el panel de hosting no permite que Apache escriba, cree los ficheros
# vacíos con permiso de escritura antes del primer arranque:
touch config.php setup.log && chmod 660 config.php setup.log
En hosting compartido el propietario suele ser el mismo usuario que ejecuta PHP (FPM/suexec), por lo que config.php en 0600 funciona. Si usa un usuario distinto para Apache, ajuste a 0660 y grupo correcto.

4. Endurecimiento con .htaccess

Bloquear el acceso web a los ficheros sensibles y forzar HTTPS:

<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  # Forzar HTTPS
  RewriteCond %{HTTPS} off
  RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
</IfModule>

# Negar acceso directo a ficheros internos
<FilesMatch "^(config\.php|setup\.log)$">
  Require all denied
</FilesMatch>

# Cabeceras de seguridad básicas
<IfModule mod_headers.c>
  Header set X-Content-Type-Options "nosniff"
  Header set X-Frame-Options "SAMEORIGIN"
</IfModule>
Importante: config.php nunca contiene credenciales operativas (sólo IDs estructurales del nodo), pero igualmente debe quedar fuera del acceso web público.

5. CORS en el API CORE

La operación de planta hace fetch directo al API CORE (p. ej. https://api.omni.josepan.app). Ese dominio debe responder a las peticiones del origen del terminal:

Access-Control-Allow-Origin: https://produccion.omni.josepan.app
Access-Control-Allow-Headers: Authorization, Content-Type, Accept
Access-Control-Allow-Methods: GET, POST, OPTIONS

La fase de instalación (wizard) no requiere CORS porque pasa por el proxy setup.php (mismo origen).

6. Primer arranque (instalación guiada)

  1. Abrir https://produccion.omni.josepan.app. Al no existir config.php, arranca el asistente.
  2. Paso 1: introducir la URL base del API CORE y pulsar "Probar Conexión" (mide latencia contra /health).
  3. Paso 2: autenticar al supervisor de infraestructura y elegir la fábrica/sede de este terminal.
  4. Paso 3: mapear la ubicación de custodia temporal (origen) y la de bodega/tránsito (destino).
  5. Paso 4: "Concluir Autoinstalación" escribe config.php y recarga hacia el login operativo.

7. Entorno local XAMPP

# 1) Copiar el proyecto en htdocs (o usar un VirtualHost dedicado):
#    C:\xampp\htdocs\produccion\

# 2) httpd-vhosts.conf  (Apache\conf\extra)
<VirtualHost *:80>
  DocumentRoot "C:/xampp/htdocs/produccion"
  ServerName produccion.local
  <Directory "C:/xampp/htdocs/produccion">
    AllowOverride All
    Require all granted
  </Directory>
</VirtualHost>

# 3) C:\Windows\System32\drivers\etc\hosts
127.0.0.1   produccion.local

# 4) php.ini  ->  habilitar:  extension=curl
# 5) Reiniciar Apache y abrir http://produccion.local
El audio (WebAudio) y la vibración requieren un primer toque del usuario y, en móviles, contexto HTTPS. En http://localhost el audio funciona; la vibración puede no estar disponible en escritorio.

8. Producción: compilar Tailwind estático (opcional pero recomendado)

El index.php usa el CDN de Tailwind para desarrollo. Para planta (rendimiento y operación offline garantizada) conviene compilar un CSS estático local y reemplazar la etiqueta del CDN por el fichero servido desde el propio nodo. El CSS crítico de respaldo embebido mantiene el terminal legible aunque el CDN no cargue.

10. Re-despliegue / purga

Para reinstalar el nodo: borrar config.php y limpiar el almacenamiento del navegador (localStorage omni1004.* e IndexedDB omni1004_outbox); o invocar setup.php con action=purge y token de supervisor. En el siguiente arranque vuelve a aparecer el asistente.
